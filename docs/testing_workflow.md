# Skill 测试流程与方案 (Systematic Test Workflow)

为了确保 Skill 在 OpenClaw 部署后具备高可靠性与优良的用户体验，我们采用了一套“三层验证”架构。

## 1. 测试架构概览
本项目将“问-动-答”全链路逻辑进行了拆解，形成了从底层代码到上层交互的完整验证。

| 测试层级 | 验证目标 | 实施方法 | 成果物 |
| :--- | :--- | :--- | :--- |
| **层级 1: 脚本逻辑** | 物理 API 调用的鲁棒性 | 单元测试 (`unittest`) | 逻辑闭环 |
| **层级 2: 交互仿真** | 触发参数与 AI 语气遵循度 | 场景化 Mock (`tests/judge.py`) | `test_report.md` |
| **层级 3: 环境测试** | 网络连通性与实时 API 状态 | 终端直接执行脚本 | 实时验证 |

## 2. 交互仿真测试详解 (Simulation)
通过 `tests/judge.py` 模拟复杂的 API 响应情况（解决 API 在线测试环境不稳定的问题）。

### 核心逻辑：
- **场景注入 (Scenario Injection)**：在 `tests/test_cases.json` 中配置用户可能的不同问法。
- **Mock 服务**：注入成功的 JSON 响应、404 未找到、或 500 连接超时。
- **AI 回复模拟**：根据 PRD 预定义的规则，验证 AI 在“显式询问”与“隐式提及”时的差异化表现。

### 如何阅读报告 (`test_report.md`)：
- **场景描述 (Scenario)**：测试的具体情况。
- **AI 动作 (Action)**：确认 AI 是否抓取了正确的 `platform` 和 `slug`。
- **逻辑依据 (Reasoning)**：解释为什么 AI 会给出接下来的回答（如：因隐式触发且 API 故障而保持静默）。

## 3. 常见情况指引 (F.A.Q)
- **Q: 为什么 Kalshi 的某个测试没有报告内容？**
  - **A**: 这往往是模拟了 API 指令失败的场景。根据 PRD 规定，非直接提问且服务故障时，AI 应静默处理，以免打扰用户对话。
- **Q: 如何增加新的测试案例？**
  - **A**: 在 `test_cases.json` 中添加一个字典条目，仿写已有的格式即可，随后执行 `python3 tests/judge.py`。
